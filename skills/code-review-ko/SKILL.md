# code-review-ko

현재 브랜치의 변경사항 또는 지정 파일을 한국어로 리뷰한다.
버그·보안·성능·가독성 순으로 우선순위를 두고 실행 가능한 피드백을 제공한다.

## 언제 사용하나

- `/code-review-ko` 또는 "코드 리뷰해줘", "이 코드 봐줘"
- "버그 있는지 확인해줘", "코드 품질 확인해줘"
- 한국어 팀 환경에서 PR 리뷰 전 사전 점검

---

## Phase 0: 리뷰 대상 파악

**리뷰 범위 결정 (우선순위 순):**

1. 사용자가 파일을 명시한 경우 → 해당 파일만
2. 스테이징된 변경이 있는 경우 → `git diff --staged`
3. 현재 브랜치 전체 변경 → `git diff main...HEAD`
4. 특정 파일을 명시하지 않은 경우 → `git status`로 확인 후 질문

```bash
git diff --staged        # 스테이징 변경
git diff main...HEAD     # 브랜치 전체 변경
git diff HEAD~1          # 마지막 커밋 변경
```

리뷰 대상이 불명확하면 사용자에게 확인한다.

## Phase 1: 코드 분석

다음 순서로 분석한다:

### 1순위 — 버그 & 보안
- 런타임 오류 가능성 (null 참조, 타입 오류, 배열 범위 초과)
- 보안 취약점 (SQL 인젝션, XSS, 인증 누락, 민감 정보 노출)
- 경쟁 조건 (race condition), 데드락
- 예외 처리 누락

### 2순위 — 성능
- 불필요한 반복 연산, N+1 쿼리
- 메모리 누수 가능성
- 비효율적인 알고리즘 (O(n²) 이상이 O(n)으로 개선 가능한 경우)

### 3순위 — 가독성 & 유지보수
- 함수/변수 이름이 의도를 명확히 전달하는지
- 중복 코드 (3번 이상 반복되는 패턴)
- 단일 책임 원칙 위반 (한 함수가 너무 많은 일을 하는 경우)
- 매직 넘버·하드코딩된 문자열

### 4순위 — 스타일 & 컨벤션
- 프로젝트 기존 코드 스타일과의 일관성
- 불필요한 주석, 잘못된 주석

## Phase 2: 리뷰 보고서 작성

다음 형식으로 한국어 리뷰를 작성한다:

```markdown
## 코드 리뷰 — <파일명 또는 범위>

### 🚨 필수 수정 (버그/보안)
> 배포 전 반드시 수정해야 할 항목

- **[파일명:줄번호]** 문제 설명
  - 현재 코드: `코드 스니펫`
  - 문제: 구체적인 위험 설명
  - 제안: `수정된 코드`

### ⚡ 권장 개선 (성능/가독성)
> 개선하면 좋지만 배포 차단은 아닌 항목

- **[파일명:줄번호]** 개선 제안

### 💡 선택 제안 (스타일/정리)
> 팀 컨벤션에 따라 선택적으로 적용

- **[파일명:줄번호]** 제안 내용

### ✅ 잘된 점
> 좋은 패턴은 명시적으로 칭찬한다

- 잘 작성된 부분 언급

---
**요약:** 필수 N건 / 권장 N건 / 선택 N건
```

**피드백 작성 원칙:**
- 문제와 이유를 함께 설명 ("이렇게 하세요" → "왜냐하면 ~ 위험이 있기 때문입니다")
- 수정 코드 예시를 가능한 함께 제공
- 칭찬할 부분도 1~2개 명시 (동기 부여)
- "필수 수정"이 없으면 명시적으로 "버그/보안 이슈 없음" 표기

## Phase 3: 후속 처리 (선택)

### --fix : 수정 사항 직접 적용

사용자가 `/code-review-ko --fix` 또는 "수정 적용해줘"라고 하면:

1. Phase 2 리뷰에서 **🚨 필수 수정** 항목만 추출한다 (배포 차단 이슈 우선).
2. 각 항목을 순서대로 처리한다:
   - `파일명:줄번호`를 파악한다.
   - `Read` 도구로 해당 파일 전체를 읽어 맥락을 확인한다.
   - `Edit` 도구로 제안된 코드로 교체한다.
   - 적용 범위가 불명확하면 해당 항목은 건너뛰고 "수동 확인 필요" 목록에 남긴다.
3. 모든 항목 처리 후 `git diff` 를 실행해 변경 내용을 출력한다.
4. 사용자에게 결과를 보고한다:
   ```
   ✅ 수정 완료: N건 적용 / M건 수동 확인 필요
   git diff 확인 후 커밋하시겠습니까?
   ```
5. **⚡ 권장 개선** 항목은 사용자 요청 시에만 추가로 적용한다.

> **주의**: Edit 도구는 정확한 문자열 일치를 요구한다. 리뷰 이후 파일이 변경됐으면
> 다시 `Read` 해서 현재 코드를 기준으로 수정한다.

### --comment : GitHub PR에 인라인 코멘트 게시

사용자가 `/code-review-ko --comment` 또는 "PR 코멘트로 올려줘"라고 하면:

**사전 확인 (실패 시 중단 후 안내):**

```bash
# gh 설치 여부
command -v gh || echo "gh CLI 미설치 — brew install gh / winget install GitHub.CLI"

# 현재 브랜치에 열린 PR이 있는지 확인
gh pr view --json number,headRefOid,baseRefName 2>/dev/null || echo "열린 PR 없음"

# owner/repo 확인
gh repo view --json owner,name
```

**코멘트 게시 절차:**

```bash
# 변수 추출
PR_NUMBER=$(gh pr view --json number --jq '.number')
COMMIT_ID=$(gh pr view --json headRefOid --jq '.headRefOid')
REPO=$(gh repo view --json owner,name --jq '"\(.owner.login)/\(.name)"')
```

각 **🚨 필수 수정** 항목과 **⚡ 권장 개선** 항목에 대해 인라인 코멘트를 게시한다:

```bash
gh api repos/"$REPO"/pulls/"$PR_NUMBER"/comments \
  --method POST \
  --field body="**[code-review-ko]** <리뷰 내용>" \
  --field commit_id="$COMMIT_ID" \
  --field path="<파일 경로>" \
  --field line=<줄번호> \
  --field side="RIGHT"
```

모든 항목 게시 후 PR 전체 리뷰 요약을 추가한다:

```bash
gh api repos/"$REPO"/pulls/"$PR_NUMBER"/reviews \
  --method POST \
  --field body="**[code-review-ko] 검토 완료** — 필수 수정 N건 / 권장 개선 M건 / 선택 제안 P건" \
  --field event="COMMENT"
```

**에러 처리:**

| 상황 | 대응 |
|------|------|
| `gh` 미설치 | 설치 안내 후 중단 |
| 열린 PR 없음 | "PR이 없습니다. PR 번호를 알려주시면 직접 지정 가능합니다" |
| `line` 필드 오류 (파일/줄 불일치) | 해당 항목만 건너뛰고, 파일 수준 코멘트로 대체 게시 |
| API 오류 (403 등) | 오류 메시지 출력 후 나머지 항목 계속 시도 |

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| diff가 1000줄 초과 | `--stat`으로 범위 확인 후 "전체/핵심 파일만" 선택지 제시 |
| 리뷰 대상 파일 없음 | `git status` 출력 후 대상 지정 요청 |
| 바이너리/생성 파일 포함 | 해당 파일 제외하고 텍스트 파일만 리뷰 |
| 테스트 파일만 변경 | 테스트 품질(커버리지, 엣지 케이스) 중심으로 리뷰 |

## 예시

**입력:** 사용자가 "코드 리뷰해줘"라고 말한 경우  
**동작:**
1. `git diff --staged` 또는 `git diff main...HEAD`로 변경사항 파악
2. 버그/보안 → 성능 → 가독성 → 스타일 순으로 분석
3. 한국어 리뷰 보고서 출력

**입력:** 사용자가 "auth.js 파일 봐줘"라고 말한 경우  
**동작:**
1. `auth.js` 파일 전체 읽기
2. 인증/보안 관련 이슈 우선 집중 분석
3. 한국어 리뷰 보고서 출력
