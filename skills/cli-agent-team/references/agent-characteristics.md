# 에이전트 특성 가이드 — 토큰 경제 + 역할 배정 기준

오케스트레이터(Claude)가 어떤 작업을 어느 에이전트에 배정할지 결정할 때 참고한다.
구체적인 숫자(rate limit, 정확한 토큰 수)는 공급사 정책에 따라 변하므로,
**상대적 특성**과 **전략적 원칙** 위주로 읽는다.

---

## 에이전트별 특성

### Claude Code (메인 오케스트레이터)

| 항목 | 내용 |
|------|------|
| 기반 모델 | Anthropic Claude (Sonnet/Opus 계열) |
| 토큰 리셋 | 사용량 기반, 약 5시간마다 리셋 (Pro 기준 — 정책 변동 있음) |
| 컨텍스트 | 200K tokens (대화 누적 포함) |
| 강점 | 복잡한 추론, 계획·검토·판단, 사용자 소통, 모호한 요구사항 해석 |
| 약점 | 대화가 길어질수록 컨텍스트 소모 ↑, 오케스트레이터로 오래 쓸수록 리밋 근접 |
| 비용 특성 | 메시지당 비용 높음. 오케스트레이터 역할에 최적화; 코드 구현을 직접 맡으면 토큰 낭비 |

**원칙: Claude는 오케스트레이터 역할(계획·검토·승인·커밋)에만 집중한다. 코드를 직접 수정하지 않는다.**
- 오타·1~3줄 수정도 FEEDBACK.md 작성 후 에이전트에게 재배정한다
- 유일한 예외: 모든 하위 에이전트가 리밋에 걸린 비상 상황 (최후 수단, 드문 경우)
- Claude가 코드를 직접 고치면 분석+구현이 겹쳐 토큰 소모가 2배가 되고, 모든 코드 변경이 에이전트 출처여야 감사 추적이 깔끔하다

### Codex CLI (OpenAI)

| 항목 | 내용 |
|------|------|
| 기반 모델 | OpenAI o-series / GPT-4o 계열 |
| 리셋 주기 | OpenAI rate limit 기준 (분당·일당 한도, 티어마다 다름) |
| 컨텍스트 | 모델마다 다름 (128K~200K 수준) |
| 강점 | 명확한 지시에 따른 정밀 코드 생성, 구조적 리팩토링, 테스트 작성 |
| 약점 | 대형 코드베이스 전체를 한 번에 처리하기엔 컨텍스트 한계 있음 |
| 비용 특성 | 중간 수준. 소~중형 작업에 비용 효율 좋음 |
| 헤드리스 실행 | `codex exec -C <dir> "<prompt>"` — stdout에 결과 출력됨 |
| 세션 이어가기 | `codex exec resume --last "<후속>"` — 대화 컨텍스트 보존 |

**최적 작업 유형**:
- 단일 기능 구현 (파일 2~5개 이하)
- 기존 코드 리팩토링 (범위 명확히 지정 시)
- 테스트 코드 작성
- 명세서가 잘 갖춰진 중형 작업

### Antigravity CLI — agy (Google Gemini 기반)

| 항목 | 내용 |
|------|------|
| 기반 모델 | Google Gemini (1.5 Pro / 2.0 계열) |
| 리셋 주기 | Google AI 정책 기준 (분당·일당 한도) |
| 컨텍스트 | 최대 1M tokens — 세 에이전트 중 가장 큰 컨텍스트 |
| 강점 | 대형 코드베이스 전체 분석, 긴 파일 읽기, 광범위한 리팩토링 |
| 약점 | 헤드리스 stdout 침묵 버그 (파일 쓰기는 됨 — `cli-dispatch-guide.md` 참고) |
| 비용 특성 | 무료 티어 한도 넉넉함 (변동 있음). 대형 작업 비용 효율 좋음 |
| 헤드리스 실행 | `agy --print "<msg>" --add-dir <dir> --print-timeout 20m` |
| 세션 이어가기 | `agy --continue --print "<후속>"` |
| 주의 | stdout이 비어도 작업은 완료됐을 수 있음 — 파일 산출물로 판단 |

**최적 작업 유형**:
- 대형 코드베이스 분석·탐색 (전체를 한 번에 읽어야 할 때)
- 광범위한 리팩토링 (여러 파일에 걸친 패턴 변경)
- 문서화 (README, API 문서 등)
- 오래 걸리는 탐색/조사 태스크 (`--print-timeout` 늘려서)

## 적응형 배분 프로토콜

고정 비율 대신, **실측 리밋 이벤트를 기록하고 그 데이터로 배분 창(window)을 조정**한다.
세션이 쌓일수록 각 에이전트가 언제 리밋에 걸리는지 패턴이 드러나고, 배분이 자동으로 최적화된다.

### 핵심 개념: 배정 창(window)

`window[X]` = 에이전트 X에 연속으로 배정할 수 있는 최대 작업 수 (마지막 리밋 이후 기준).

- **초기값**: 모든 에이전트 window = 3 (데이터 없을 때 보수적 기본값)
- **리밋 발생 시**: `window = max(1, 리밋까지_처리한_작업수 - 1)` → 줄임
- **연속 2세션 성공 시**: `window = min(window + 1, 10)` → 서서히 늘림
- **window 상태는 LOG.md 상단에 저장**, Claude가 배정 전마다 읽는다

예시:
```
Codex가 4번째 작업에서 리밋 → window[Codex] = 3
다음 세션에 3작업 성공 → 성공 횟수 +1 (아직 window 유지)
그 다음 세션도 3작업 성공 → window[Codex] = 4로 복원
```

### 도메인 우선순위 (고정, 조정 대상 아님)

| 작업 유형 | 1순위 | 폴백 1 | 폴백 2 |
|----------|-------|-------|-------|
| 소형 구현 (1~20줄) | Codex | agy | Claude (최후 수단) |
| 중형 구현 (20~200줄) | Codex | agy | Claude 직접 (최후) |
| 대형 구현 / 분석 (200줄↑) | agy | Codex (분할) | Claude 직접 (최후) |
| 코드 리뷰·검토 | Claude | — | — |
| 커밋/PR | Claude (항상) | — | — |

> 도메인 1순위가 window 초과 상태이면 다음 우선순위로 내려간다.
> window 내라도 연속 배정이 너무 치우치면 로테이션을 고려한다.

### 배정 전 결정 절차 (Phase 5 단계 1.5)

```
1. LOG.md 상단의 "현재 배정 창" 표 읽기
2. 작업 유형으로 1순위 에이전트(E) 결정
3. E의 "현재 세션 작업 수" < window[E] → E 배정
4. 초과 상태 → 다음 우선순위 에이전트로 이동
5. 모두 초과 → 도메인 1순위 쓰되 LOG.md에 "창 초과 배정" 기록
```

### 리밋 발생 시 처리

1. LOG.md에 ⚠️ 이벤트 기록 (타임스탬프, 에이전트, 그 세션의 작업 수)
2. `window[에이전트] = max(1, 세션작업수 - 1)` 로 업데이트
3. 폴백 에이전트에 같은 TASK.md 재배정
4. 모든 폴백 실패 → PLAN.md "보류" + 사용자 알림
5. **Claude 리밋**: 오케스트레이터 대체 불가 → 사용자에게 세션 재시작 안내

### Claude 토큰 보존 원칙

Claude 리밋이 가장 치명적이므로(오케스트레이터 역할 대체 불가):

- 검토·승인·커밋 외에 직접 코드 구현하지 않는다
- 에이전트 결과물은 `git diff`로만 검토 (전체 파일 읽기 자제)
- 대화가 길어지면 `/compact` 또는 새 세션으로 이어간다 (`PLAN.md`가 상태 보존)
