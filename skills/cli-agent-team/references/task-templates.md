## 디렉토리 구조

```
_agent_reports/
  LOG.md                      ← 전체 작업 이력 한 줄 로그
  <task-id>/
    TASK.md                   ← 오케스트레이터 작성, 에이전트가 가장 먼저 읽음
    REVIEW.md                 ← 에이전트가 검토 단계에서 작성 (실행 전 의견)
    TODO.md                   ← 에이전트가 실행 착수 직후 작성, 진행하며 갱신
    REPORT.md                 ← 에이전트가 완료 후 작성
    FEEDBACK.md               ← (반복 시) 오케스트레이터가 추가 수정 요청 적음
  <task-id>-CONTRACT.md       ← (두 에이전트가 맞물리는 작업일 때만) 공유 계약
```

**파일 생성 순서**: TASK.md → REVIEW.md → (승인 후) TODO.md → REPORT.md

## TASK.md (오케스트레이터가 작성)

```markdown
# <task-id> 작업 지시

작업 디렉토리: <절대경로>
담당: <에이전트 이름>
배정일: <날짜>

## 시작 전 필독
- CLAUDE.md / 프로젝트 컨벤션 문서
- PLAN.md
- AGENT_ROLES.md 의 안전장치 규칙 + REPORT.md 형식

## 담당 영역 / 패턴
<이 에이전트가 지켜야 할 프로젝트 고유 컨벤션 — 예: 단일 출처 유틸 위치, 코드 스타일,
디렉토리 책임 분리>

## 진행 방식
1. 착수 직후 TODO.md에 하위작업 체크리스트를 작성한다.
2. 진행하며 체크박스를 갱신한다. 막히면 "⚠️ 막힘: <이유>"를 적는다.
3. 완료 후 REPORT.md를 작성한다.

## 구체적 작업 지시
<배경, 범위, 기대 결과물을 구체적으로 — 파일/라인 단위로 구체적일수록 결과 품질이 높다>

## 완료 기준 (Acceptance Criteria)
<!-- verify.sh가 REPORT.md의 AC 체크리스트와 대조한다. 검증 가능하고 구체적으로 작성할 것 -->
- [ ] <기능 기준 — 예: GET /api/users 호출 시 200과 사용자 배열을 반환한다>
- [ ] <품질 기준 — 예: 기존 테스트가 모두 통과한다>
- [ ] 허용 파일 외 파일을 건드리지 않았다

## 허용 파일 (verify.sh가 스코프 검사에 사용)
<!-- 건드려도 되는 파일 또는 디렉토리를 한 줄에 하나씩. 디렉토리는 접두사 매칭됨 -->
- <예: src/api/users.ts>
- <예: tests/api/>

## 완료 증거 파일 (verify.sh가 존재 여부를 확인)
<!-- 작업 완료 시 반드시 존재하거나 변경되어야 하는 파일. "수정됨" 또는 "생성됨" 중 택일 -->
<!-- agy처럼 stdout이 침묵해도 파일 산출물로 실제 완료를 판단할 수 있다 -->
- <예: src/api/users.ts  수정됨>
- <예: tests/api/users.test.ts  생성됨>

## 금지 파일
<!-- 이번 작업에서 건드리지 말 것 — 이유를 함께 적으면 에이전트가 의도를 이해함 -->
- <예: src/auth/  (M2에서 별도 처리 예정)>
```

**모든 TASK.md에 고정 포함할 안전장치 규칙**:
- 이 파일은 **검토 단계**와 **실행 단계**로 구분 실행된다. 디스패치 메시지를 따른다:
  - `review` 메시지를 받으면 → REVIEW.md만 작성, 코드·파일 건드리지 않음
  - `execute` 메시지를 받으면 → TODO.md + 실제 작업 + REPORT.md 작성
- git commit/push 금지 (오케스트레이터가 검토 후 처리)
- 지시된 범위 밖 파일을 건드리지 않는다
- 민감 파일(.env, 토큰, 키 등)을 읽거나 출력하지 않는다
- 기존 패턴/컨벤션을 따른다 (새 추상화·의존성 임의 추가 금지)

## REVIEW.md (에이전트가 작성 — 검토 단계)

실행 전 에이전트가 BRIEF.md + TASK.md를 읽고 의견을 작성한다. 코드를 건드리지 않는다.
Claude가 이 파일을 읽고 사용자에게 요약 보고 → 승인 후 실행 단계로 진행한다.

```markdown
# <task-id> 검토 의견 (<agent-name>)

## 이해한 내용
(BRIEF.md와 TASK.md를 읽고 이렇게 이해했습니다 — 틀렸거나 빠진 게 있으면 정정 요청)

## 우려사항 / 위험 요소
- (없으면 "없음")

## 제안 / 대안
(더 나은 접근 방법이 있다면 — 없으면 "없음")

## 실행 전 확인이 필요한 질문
- (없으면 "없음")

## 예상 작업 범위
- 건드릴 파일/함수: ...
- 건드리지 않을 것: ...
```

오케스트레이터(Claude)가 REVIEW.md를 보고:
- **우려사항·제안이 타당하면** TASK.md를 수정하고 재검토를 요청한다
- **질문이 있으면** 사용자에게 묻고 TASK.md에 답을 추가한다
- **문제없으면** 실행 단계(`dispatch.sh ... execute`)로 진행한다

```markdown
# <task-id> 작업 보고서 (<agent-name>)

## 요약
(무엇을, 왜)

## 변경 파일
- path/to/file — 변경 내용 한 줄 요약

## AC 체크리스트 (TASK.md 완료 기준 확인)
<!-- verify.sh가 이 섹션을 파싱한다. [ ] → 미완료, [x] → 완료로 정확히 표기할 것 -->
- [x] <TASK.md의 완료 기준 1을 그대로 복사 후 완료 여부 체크>
- [x] <완료 기준 2>

- 실행한 명령과 결과 (lint/build/test 등 — 실제 출력 일부 포함)

## 의견 및 제안
(작업하면서 발견한 더 나은 대안, 기존 설계의 문제점, 추천 사항. 지시 범위를 넘어서는
통찰이라도 여기 적는다 — 코드로 구현하지는 말고 의견만. 없으면 "없음")

## 질문 / 막힌 점
(오케스트레이터나 사용자의 확인이 필요한 것. 없으면 "없음")

## 미해결 이슈
(시간/범위 제약으로 처리하지 못한 것. 없으면 "없음")
```

오케스트레이터는 매번 "의견 및 제안"과 "질문/막힌 점"을 반드시 읽는다 — 구조적 사안이면
PLAN.md "보류/미정"으로 옮기고 사용자에게 묻고, 범위 내 기술 사안이면 직접 채택/반려를
판단해 사유를 한 줄 남긴다.

## FEEDBACK.md (오케스트레이터 Claude가 작성 — 재배정 시)

Claude는 코드를 직접 수정하지 않는다. 규모와 관계없이 항상 FEEDBACK.md를 작성해
에이전트에게 재배정한다. Claude의 역할은 "무엇을 어떻게 고쳐야 하는가"를 명확히
기술하는 것이고, 실제 코드 수정은 에이전트가 담당한다.

```markdown
# <task-id> 피드백 (회차 N)

## 검토 기준
- 참고한 것: BRIEF.md의 [인용], TASK.md의 완료 기준 [인용]

## 지적 사항
- 사항 1: `src/foo.js:42` — 무엇이 문제인가 / 왜 의도와 어긋나는가
- 사항 2: `src/bar.js:15-20` — ...

## 수정 지시
각 사항에 대해 에이전트가 판단 없이 실행할 수 있을 만큼 구체적으로 기술한다:
- 사항 1 수정 방법: ...
- 사항 2 수정 방법: ...

## 범위 제한 (건드리지 말 것)
- 이번 수정에서 건드리지 않아야 할 파일/함수/로직을 명시

## 완료 기준
- [ ] 수정 후 확인해야 할 것
```

**재배정 방식 (SKILL.md Phase 5 단계 8 참고)**:
- 세션 이어가기 지원 CLI → `resume` / `--continue` 플래그 우선 (컨텍스트 유지 효율적)
- 세션 만료 시 → `dispatch.sh <cli> <task-id> <auth-mode> [dir] feedback`
- 재배정 3회 초과 시 → 작업 보류 후 사용자에게 보고

## CONTRACT.md (두 에이전트가 맞물릴 때)

두 에이전트가 같은 기능의 다른 절반(예: 서버/클라이언트)을 동시에 작업할 때, 인터페이스를
먼저 고정해서 양쪽이 서로의 완성을 기다리지 않고 병렬로 진행하게 한다.

```markdown
# <feature> 계약 (<agent-A>/<agent-B> 공통 참조)

## 인터페이스
<이벤트명/함수 시그니처/API 엔드포인트 등 — 양쪽이 합의해야 하는 것만 최소한으로>

## 폴백/에러 정책
<한쪽이 실패하거나 안 붙어 있어도 다른 쪽이 깨지지 않는 방법>

## 배경 문서
이 계약은 <설계 조사 보고서 경로>를 기반으로 확정되었다. 구현 중 이견이 있으면 비교해
더 나은 쪽을 택하고 본인 REPORT.md에 사유를 남긴다.
```

## LOG.md (오케스트레이터가 관리 — 적응형 배분의 핵심)

`_agent_reports/LOG.md`는 두 역할을 동시에 한다:
1. **현재 배정 창(window) 상태** — 배정 전마다 Claude가 읽고 결정에 사용
2. **전체 이벤트 이력** — 리밋 패턴 파악, 사후 분석

```markdown
# Agent LOG — <프로젝트명>

## 현재 배정 창 (Claude가 배정마다 업데이트)

| 에이전트 | window | 현재 세션 작업 수 | 연속 성공 세션 | 마지막 리밋 | 상태 |
|----------|--------|----------------|------------|-----------|------|
| codex    | 3      | 0              | 0          | —         | 정상 |
| agy      | 3      | 0              | 0          | —         | 정상 |

**window 조정 규칙:**
- 리밋 발생 시: `window = max(1, 세션작업수 - 1)` / 현재 세션 작업 수 → 0 리셋
- 세션 성공 시: 연속 성공 세션 +1 / 현재 세션 작업 수 → 0 리셋
- 연속 성공 세션 = 2 도달 시: `window = min(window + 1, 10)` / 연속 성공 세션 → 0 리셋

---

## 이벤트 이력

| 날짜시간 | 작업ID | 에이전트 | 규모 | 결과 | 세션 작업 수 | 메모 |
|---------|-------|-------|-----|-----|------------|-----|
| 2024-01-15 10:00 | T001 | codex | 중형 | ✅ 완료 | 1 | |
| 2024-01-15 12:30 | T002 | agy   | 대형 | ✅ 완료 | 1 | |
| 2024-01-15 14:00 | T003 | codex | 중형 | ✅ 완료 | 2 | |
| 2024-01-15 15:30 | T004 | codex | 중형 | ⚠️ 리밋  | 3 | window: 3→2 |
| 2024-01-15 15:35 | T004 | agy   | 중형 | ✅ 완료 | 2 | 폴백 처리 |
```

**이벤트 결과 기호:**
- `✅ 완료` — 정상 종료, REPORT.md 확인됨
- `⚠️ 리밋` — 리밋/quota 감지, window 조정 필요
- `❌ 오류` — 리밋 외 오류 (로그 확인 필요)
- `🔁 폴백` — 리밋으로 인한 대체 에이전트 처리
- `⏭️ 창 초과` — window 초과였으나 대안 없어 배정한 경우

**세션 정의:** 리밋 발생 → 다음 리밋 발생 사이의 연속 작업들.
리밋이 없으면 프로젝트 시작부터 현재까지가 하나의 세션.

## 병렬 검증 (중요/애매한 작업)

같은 작업을 두 에이전트에게 동시에 시켜 비교 채택할 때는 git worktree로 격리한다 (같은
파일을 동시에 건드려 충돌하는 것을 방지):

```bash
git worktree add ../<repo>-A-wt -b agent/A-<task-id> HEAD
git worktree add ../<repo>-B-wt -b agent/B-<task-id> HEAD
# 각 worktree 절대경로를 TASK.md의 "작업 디렉토리"로 명시
# 완료 후 비교:
git -C ../<repo>-A-wt diff HEAD
git -C ../<repo>-B-wt diff HEAD
# 채택한 쪽만 메인 워킹트리에 적용 후 정리:
git worktree remove ../<repo>-A-wt --force
git worktree remove ../<repo>-B-wt --force
```

TASK.md에 다음을 추가한다: "이 작업은 동일한 지시가 다른 에이전트에게도 동시에
배정되었습니다. 서로의 결과를 참고하거나 조율하지 말고 독립적으로 최선의 구현을
작성하세요."

## AGENT_ROLES.md — 자동 검증 명령어 섹션

`verify.sh`가 AGENT_ROLES.md에서 이 섹션을 읽어 자동 실행한다.
프로젝트 설정에 따라 AGENT_ROLES.md에 아래 형식으로 추가한다:

```markdown
## 자동 검증 명령어
<!-- verify.sh가 Phase 5 단계 6마다 이 명령어들을 순서대로 실행한다 -->
<!-- 형식: <레이블>: <실행할 명령어> (프로젝트 루트 기준) -->
lint: npm run lint
test: npm test
build: npm run build
```

명령어가 없으면 섹션 자체를 생략한다 — verify.sh가 건너뜀.
프로젝트 언어에 따라 명령어를 조정한다:
- Python: `test: pytest`, `lint: ruff check .`
- Go: `test: go test ./...`, `lint: go vet ./...`
- Rust: `test: cargo test`, `lint: cargo clippy`
