관련: [[soul-template]] | [[task-templates]] | [[../SKILL.md]]

이 파일은 프로젝트 루트의 `AGENTS.md`로 복사해 사용하는 사람 큐레이션 지식 템플릿이다.
Memory MCP에 자동 저장되는 기억과 달리, 이 문서는 사람이 검토하고 승인한 프로젝트 운영 규칙만 담는다.

---

## 에이전트 행동 원칙

### 1. Think Before Coding
- 작업 시작 전에 목표, 가정, 허용 범위, 성공 기준을 명시한다.
- 요구사항이 모호하거나 위험한 변경이 예상되면 코딩 전에 질문한다.
- 확인되지 않은 추측을 사실처럼 다루지 않는다.

### 2. Simplicity First
- 요청받은 것만 최소한으로 구현한다.
- 미래에 필요할 수 있다는 이유만으로 기능, 옵션, 추상화를 추가하지 않는다.
- 기존 구조로 충분하면 새 구조를 만들지 않는다.

### 3. Surgical Changes
- 태스크와 무관한 코드, 주석, 포맷, 파일명을 건드리지 않는다.
- 변경 범위를 좁게 유지하고, 기존 패턴을 우선 따른다.
- 리팩토링은 명시적으로 요청되었거나 문제 해결에 필수일 때만 수행한다.

### 4. Goal-Driven Execution
- 테스트 가능한 완료 기준을 먼저 확인한다.
- 구현 후에는 가능한 검증 명령을 실행하고 결과를 기록한다.
- 검증할 수 없는 항목은 이유와 남은 리스크를 보고한다.

## 에이전트 역할 카드

### codex

| 항목 | 내용 |
|------|------|
| 강점 | 알고리즘 구현, 단위 테스트, 성능 최적화, 복잡한 리팩토링 |

| 모델 티어 | quality |
| 적합한 작업 | 복잡도 높은 구현, 테스트 가능한 코드 변경, 성능 병목 개선 |

### agy

| 항목 | 내용 |
|------|------|
| 강점 | 인라인 수정, 문서 작성, 타입 정의, 단순 반복 작업 |

| 모델 티어 | fast |
| 적합한 작업 | TODO 갱신, 문서 정리, 주석 정리, 반복적인 소규모 변경 |

## 모델 티어 선택 기준

| 티어 | 예시 모델 | 사용 기준 |
|------|-----------|-----------|
| fast | gpt-5.4-mini, claude-haiku | TODO 갱신, 문서 작성, 주석 정리, REPORT 작성, 단순 반복 작업 |
| quality | gpt-5.5, claude-sonnet | 인증·보안, DB 스키마, 새 API, 복잡한 비즈니스 로직, 성능 최적화 |

원칙:
- 실패 비용이 낮고 반복성이 높은 작업은 fast 티어를 사용한다.
- 보안, 데이터 무결성, 공개 API, 복잡한 설계 판단이 포함되면 quality 티어를 사용한다.
- 어떤 티어를 선택했는지와 이유를 태스크 로그에 남긴다.

## 아키텍처 결정 로그

중요한 구조 결정은 아래 형식으로 기록한다.

| 날짜 | 결정 내용 | 이유 |
|------|-----------|------|
| YYYY-MM-DD | 예: 인증 로직은 기존 미들웨어를 확장한다 | 새 인증 경로를 만들면 정책 중복과 누락 위험이 커진다 |

기록 기준:
- 모듈 경계, 데이터 흐름, 저장소 구조, API 계약이 바뀌는 결정은 기록한다.
- 단순 구현 세부사항이나 임시 디버깅 내용은 기록하지 않는다.

## 에이전트 배분 원칙

scopir.com 패턴을 기준으로 작업 성격에 맞게 배분한다.

| 작업 유형 | 우선 에이전트 | 이유 |
|-----------|---------------|------|
| 다중 파일 리팩토링 | Claude | 넓은 문맥에서 판단과 검토가 필요하다 |
| 알고리즘 구현 | codex | 정밀 구현, 테스트 작성, 성능 개선에 강하다 |
| 인라인 수정 | agy | 좁은 범위의 반복 수정과 문서 작업에 적합하다 |

배분 전 확인:
- 허용 파일 목록과 금지 파일 목록을 먼저 확인한다.
- 아키텍처 판단이 필요한 작업은 agy 단독으로 진행하지 않는다.
- 실패가 반복되면 같은 방식으로 재시도하지 말고 실패 패턴에 기록한 뒤 접근을 바꾼다.

## 반복 실패 패턴

동일한 실패가 반복되면 아래 형식으로 기록한다.

| 태스크ID | 증상 | 원인 | 해결책 |
|----------|------|------|--------|
| T000 | 예: 테스트가 로컬에서만 통과하고 CI에서 실패 | 환경변수 기본값 차이 | CI 설정을 먼저 확인하고 테스트에서 명시적으로 주입한다 |

기록 기준:
- 같은 오류가 2회 이상 반복되면 기록한다.
- 원인이 불명확하면 추정 원인과 확인할 검증 방법을 함께 적는다.
- 해결 후에는 재발 방지 규칙을 추가한다.

## 태스크 완료 기준

공통 AC 체크리스트 예시:

- [ ] 요청한 기능 또는 문서 변경이 완료됨

- [ ] 기존 동작과 호환성이 유지됨
- [ ] 관련 테스트 또는 검증 명령을 실행함
- [ ] 실행하지 못한 검증이 있다면 이유를 보고함
- [ ] 완료 보고서에 변경 내용과 검증 결과를 기록함
