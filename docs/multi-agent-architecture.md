# 다중 에이전트 아키텍처 — 개념과 구조

> 이 문서는 실제 Tarkov Market KO 프로젝트에서 검증된 패턴을 범용화한 것이다.
> 새 프로젝트에 적용할 때는 `docs/project-setup-guide.md`를 참고한다.

---

## 왜 다중 에이전트인가

단일 Claude 세션은 긴 작업에서 컨텍스트가 오염된다.  
에이전트를 분리하면:

- 각 에이전트가 **자신의 도메인만** 집중 — 컨텍스트 오염 없음
- 병렬 실행 가능 (UI 개발 + 백엔드 개발 동시)
- 역할이 명확해서 **책임 추적**이 쉬움
- 오케스트레이터가 검증 후 커밋 — 에이전트가 직접 git push 안 함

---

## 핵심 구성 요소

```
┌─────────────────────────────────────────┐
│            오케스트레이터 (Claude)         │
│  - 사용자와 대화                           │
│  - 에이전트 배정 / 결과 검증 / git 커밋     │
│  - PLAN.md 관리                          │
└────────┬──────────────┬─────────────────┘
         │              │
    ┌────▼────┐    ┌────▼────┐
    │ planner │    │qa-analyst│
    │ 설계    │    │ 검증    │
    └────┬────┘    └────▲────┘
         │              │
    ┌────▼──────────────┴────┐
    │  구현 에이전트들         │
    │  (ui-dev, bot-dev, ...) │
    └────────────────────────┘
```

### 오케스트레이터

Claude 본체(사용자와 대화하는 메인 세션).  
역할:
- 사용자 요청을 받아 워크플로우 결정
- `Agent()` 도구로 서브 에이전트 생성
- 각 에이전트 결과물을 직접 읽고 검증
- `git add / commit` 담당 (에이전트는 절대 커밋 안 함)

### planner (설계 에이전트)

모든 기능 개발의 첫 번째 단계.  
코드베이스를 읽고 설계서를 작성한다.

- **입력**: 사용자 요청
- **출력**: `_workspace/01_plan_{feature}.md`
- **핵심**: 추정 금지, 실제 파일을 읽고 작성

### 구현 에이전트 (예: ui-dev, bot-dev)

planner 설계서를 받아 실제 코드를 작성.

- **입력**: `_workspace/01_plan_{feature}.md`
- **출력**: 실제 파일 수정 + `_workspace/02_{name}_{feature}_done.md`
- **핵심**: 설계서 범위 밖 수정 금지

### qa-analyst (검증 에이전트)

구현 결과를 독립적으로 검증.

- **입력**: 설계서 + 구현 완료 보고서
- **출력**: `_workspace/03_qa_{feature}_report.md` (PASS/FAIL/PARTIAL)
- **핵심**: 코드를 직접 수정하지 않음 — 수정 지침만 작성

## 통신 패턴

### 파일 기반 (주요 방식)

```
_workspace/
├── 00_input.md          # 사용자 원본 요청
├── 01_plan_{feature}.md # planner 설계서
├── 02_ui_{feature}_done.md   # ui-dev 완료 보고서
├── 02_bot_{feature}_done.md  # bot-dev 완료 보고서
└── 03_qa_{feature}_report.md # qa-analyst 검증 결과
```

파일명의 번호가 실행 순서를 나타낸다.

### SendMessage (실시간 알림)

에이전트가 완료 시 오케스트레이터에게 알림:

```
SendMessage → 오케스트레이터:
  "설계서 완료: _workspace/01_plan_xxx.md"
```

에이전트 간 협의가 필요할 때:

```
ui-dev → bot-dev: "이 API 엔드포인트 스펙 확인 필요"
bot-dev → ui-dev: "GET /api/items 응답 shape: { items: [...] }"
```

## 워크플로우 유형

### A. 단일 도메인 기능 개발

```
오케스트레이터
  → planner (설계서 작성)
  → ui-dev 또는 bot-dev (구현)
  → qa-analyst (검증)
  → 오케스트레이터 (검토 후 커밋)
```

### B. 풀스택 기능 개발

```
오케스트레이터
  → planner (설계서 작성)
  → ui-dev + bot-dev [병렬 실행]
      (API 스펙은 SendMessage로 조율)
  → qa-analyst (경계면 교차 검증)
  → 오케스트레이터 (검토 후 커밋)
```

### C. 버그 픽스

```
오케스트레이터
  → qa-analyst (원인 분석 + 수정 지침)
  → ui-dev 또는 bot-dev (수정 적용)
  → 오케스트레이터 (검토 후 커밋)
```

### D. 퀄리티 감사

```
오케스트레이터
  → qa-analyst (전체 감사 → quality_matrix.md)
  → 오케스트레이터 (매트릭스 파싱 → 우선순위 정렬)
  → ui-dev / bot-dev 순차 실행 (항목별)
  → 오케스트레이터 (각 항목 커밋)
```

## 에이전트 파일 형식

```markdown
---
name: agent-name
description: "한 줄 설명 — 오케스트레이터가 언제 이 에이전트를 부를지 판단 기준"
tools:
  - Read
  - Edit
  - Write
  - Bash
  - Glob
  - Grep
---

# Agent Name

## 핵심 역할
(이 에이전트가 담당하는 도메인과 책임 범위)

## 프로젝트 컨텍스트
(이 에이전트가 알아야 할 프로젝트 특화 정보)

## 작업 원칙
(에이전트가 따라야 할 규칙)

## 입력/출력 프로토콜
(무엇을 받고 무엇을 만들어내는지)

## 에러 핸들링
(예외 상황별 대응)

## 팀 통신 프로토콜
(SendMessage 수발신 대상)
```

## 오케스트레이터 스킬 형식

```markdown
---
name: my-project-dev
description: "이 프로젝트의 모든 개발 작업을 에이전트 팀으로 조율한다."
---

# My Project Orchestrator

## 실행 모드 결정
(워크플로우 유형 → 팀 구성 매핑 표)

## 워크플로우
Phase 0: 컨텍스트 확인 (_workspace/ 존재 여부)
Phase 1: 준비 (_workspace/ 생성, 입력 저장)
Phase 2: 워크플로우별 에이전트 실행
Phase 3: 결과 수집 + 사용자 보고
Phase 4: 커밋 안내
```

## 핵심 규칙 요약

| 규칙 | 이유 |
|------|------|
| 에이전트는 git commit 하지 않음 | 오케스트레이터가 검증 후 커밋해야 안전 |
| planner는 추정하지 않음 | 실제 파일을 읽어야 정확한 설계 가능 |
| qa-analyst는 코드 수정 안 함 | 분리된 검증 역할 유지 |
| _workspace/ 번호 순서 | 실행 순서를 파일명으로 표현 |
| 긴 prefix → 짧은 prefix 순서 | startsWith 오탐 방지 (봇 명령어 라우터) |
